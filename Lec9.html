Algorithm Evaluation คืออะไร — ภาพรวม
concept
Algorithm Evaluation คือการประเมินว่า algorithm ที่ออกแบบมานั้นดีแค่ไหน โดยดูจากการใช้ทรัพยากร 2 อย่างหลัก คือ เวลา และ หน่วยความจำ
เป้าหมายคือนำไปเปรียบเทียบกับ algorithm อื่น เพื่อเลือกตัวที่ดีที่สุดสำหรับงานนั้น ๆ
2 ระยะที่ต้องรู้
📐 A Priori Estimates — การวิเคราะห์สมรรถนะ
ทำก่อนรันโปรแกรม หลังจากออกแบบ algorithm แล้ว
วิเคราะห์จาก pseudo-code หรือตัวโปรแกรม
อิสระจากสภาพแวดล้อม (hardware/software)
🧪 A Posterior Testing — การวัดสมรรถนะ
ทำหลังจาก implement โปรแกรมแล้ว ทดสอบจริง
เปรียบเทียบโปรแกรมที่พัฒนาจาก algorithm ต่างกัน
ขึ้นอยู่กับ hardware, software, network, ข้อมูลที่ใช้ทดสอบ
💡 ง่าย ๆ: A Priori = วิเคราะห์บนกระดาษก่อนเขียนโค้ด | A Posterior = รันจริงแล้ววัดเวลา
2
Space Complexity — ความซับซ้อนเชิงพื้นที่
concept
Space Complexity คือจำนวนหน่วยความจำที่โปรแกรมใช้ในการทำงาน นอกเหนือจากหน่วยความจำที่ใช้เก็บข้อมูลเข้า
สูตร: S(P) = c + s(n) โดย c คือส่วนที่คงที่ และ s(n) คือส่วนที่แปรผัน
🔒 Fixed Part [c]
อิสระจากขนาดข้อมูล n เช่น ตัวแปรทั่วไปที่ประกาศไว้ ไม่ว่า n จะเป็นเท่าไหร่ ส่วนนี้ก็เท่าเดิม
📈 Variable Part [s(n)]
ขึ้นอยู่กับขนาดข้อมูล n เช่น array ขนาด n หรือ recursive stack ที่ลึกตาม n
ตัวอย่างที่ 1 — ฟังก์ชันธรรมดา
def abc(a, b, c): return (a + b + b*c +(a+b-c)/(a+b) + 4.0) abc(1, 2, 3)
Fixed [c]
= 3 หน่วย (a, b, c)
Variable [s(n)]
= 0 (ไม่มี array หรือ recursion)
S(P)
>= 3 หน่วย คงที่เสมอ
ตัวอย่างที่ 2 — วนลูปกับ List
def sum(given_list): s = 0 for i in range(0, len(given_list)): s = s + given_list[i] return s
Fixed [c]
>= 3 (s, i, len)
Variable [s(n)]
>= n (given_list)
S(P)
>= n + 3
ตัวอย่างที่ 3 — Recursive Function (สำคัญ!)
def rsum(given_list, length): if length <= 0: return 0 return rsum(given_list, length-1) + given_list[length-1]
Recursive Stack ต้องการพื้นที่ต่อ 1 รอบ: Formal param (1) + Local var (1) + Return address (1) = 3 หน่วย
Depth of Recursion = n+1 → s(n) = 3(n+1)
Fixed [c]
มีบ้าง (ขึ้นกับ impl)
Variable [s(n)]
= 3(n+1)
S(P)
>= 3(n+1)
⚠️ จำไว้: ฟังก์ชัน Recursive จะกินพื้นที่เพิ่มขึ้นตาม ความลึกของการเรียกซ้ำ เพราะต้องเก็บ state ของแต่ละรอบไว้ใน call stack
3
Time Complexity — ความซับซ้อนเชิงเวลา
concept
T(P) ของโปรแกรมประกอบด้วย compile time (อิสระจาก n) และ run time t(n) (ขึ้นกับ n)
เวลาพูดถึง time complexity ในวิชานี้ หมายถึง run time t(n) เสมอ
วิธีวิเคราะห์: นับจำนวน statement / step / operations ที่โปรแกรมต้องทำ (ไม่ได้จับเวลาจริง)
🧠 ทำไมไม่จับเวลาจริง? เพราะเวลาจริงขึ้นอยู่กับ hardware — คอมเร็วก็เร็ว คอมช้าก็ช้า วิเคราะห์จากจำนวน step แทน ก็จะ hardware-independent
ตัวอย่างที่ 1 — ฟังก์ชันธรรมดา
def abc(a, b, c): return (a + b + b*c +(a+b-c)/(a+b) + 4.0) # 1 step เท่านั้น
t(n) = 1 — มีแค่ return statement เดียว ไม่ว่าข้อมูลจะมีกี่ตัว ก็ทำ 1 step เสมอ
ตัวอย่างที่ 2 — วนลูป
def sum(given_list): s = 0 # 1 step (count for s) for i in range(0, len(given_list)): # n+1 steps (for loop) s = s + given_list[i] # n steps (assignment) return s # 1 step (return)
t(n) = 1 + (n+1) + n + 1 = 2n + 3 — ยิ่ง n มาก ยิ่งใช้เวลามาก แบบ linear
ตัวอย่างที่ 3 — Recursive (สำคัญ!)
def rsum(given_list, length): if length <= 0: return 0 # count for "if" และ "return" return rsum(given_list, length-1) + given_list[length-1]
กรณี n=0: t(0) = 2
กรณี n>0: t(n) = 2 + t(n-1)
แก้ recurrence relation:
t(n) = 2 + t(n-1)
       = 2 + 2 + t(n-2)
       = 2 + 2 + 2 + ... + t(0)
       = 2n + t(0) = 2n + 2
→ t(n) = 2n + 2

Asymptotic Notations — สัญกรณ์เชิงกำกับ
ทำไมต้องมี?
เราไม่ได้อยากรู้ค่า t(n) แบบ exact เช่น "2n+3 steps" เราอยากรู้ แนวโน้ม เมื่อ n ใหญ่มาก ๆ
เช่น 2n+3 กับ 100n+6 ต่างกันในรายละเอียด แต่ทั้งคู่เติบโตแบบ linear เหมือนกัน → จัดกลุ่มเป็น O(n) ได้
ผลการวิเคราะห์แสดงในรูป Asymptotic Notation: Big-O, Omega, Theta, Little-o, Little-omega
O
Big-O — ขอบเขตบน (Upper Bound)
f(n) ≤ c·g(n) สำหรับทุก n ≥ n₀
f(n) = O(g(n)) ถ้ามีค่าคงที่ c และ n₀ ที่เป็นบวก ทำให้ f(n) ≤ c·g(n) สำหรับทุก n ≥ n₀
แปลไทย: g(n) คือ เพดานบน ของ f(n) — f(n) จะโตได้ไม่เกิน g(n) (คูณด้วยค่าคงที่)
ตัวอย่าง: 3n+3 = O(n) เพราะ 3n+3 ≤ 4n สำหรับ n ≥ 3 | 10n²+4n+2 = O(n²) เพราะ 10n²+4n+2 ≤ 11n² สำหรับ n ≥ 5
ข้อสังเกต: 3n+3 = O(n²) ก็ได้ (upper bound ไม่จำเป็นต้องแน่น) | แต่ 3n+3 ≠ O(1)
Ω
Omega — ขอบเขตล่าง (Lower Bound)
f(n) ≥ c·g(n) สำหรับทุก n ≥ n₀
f(n) = Ω(g(n)) ถ้ามีค่าคงที่ c และ n₀ ที่เป็นบวก ทำให้ f(n) ≥ c·g(n) สำหรับทุก n ≥ n₀
แปลไทย: g(n) คือ พื้นล่าง ของ f(n) — f(n) จะโตเร็วกว่า g(n) เสมอ (อย่างน้อย)
ตัวอย่าง: 3n+3 = Ω(n) เพราะ 3n+3 ≥ 3n สำหรับ n ≥ 1 | 3n+3 = Ω(1) เพราะ 3n+3 ≥ 1 สำหรับ n ≥ 1
Θ
Theta — ขอบเขตแน่น (Tight Bound)
c₁·g(n) ≤ f(n) ≤ c₂·g(n) สำหรับทุก n ≥ n₀
f(n) = Θ(g(n)) ถ้ามีค่าคงที่ c₁, c₂ และ n₀ ทำให้ c₁·g(n) ≤ f(n) ≤ c₂·g(n)
แปลไทย: g(n) เป็นทั้ง upper bound และ lower bound ของ f(n) พร้อมกัน — แม่นยำที่สุด
ตัวอย่าง: 3n+3 = Θ(n) เพราะ 3n ≤ 3n+3 ≤ 4n สำหรับ n ≥ 2 | ทฤษฎี: polynomial degree m → Θ(nᵐ)
o/ω
Little-o และ Little-omega — ขอบเขตเข้ม (Strict)
Little-o: f(n) = o(g(n)) ก็ต่อเมื่อ lim(n→∞) f(n)/g(n) = 0 → f(n) โตช้ากว่า g(n) อย่างเข้ม
Little-ω: f(n) = ω(g(n)) ก็ต่อเมื่อ lim(n→∞) g(n)/f(n) = 0 → f(n) โตเร็วกว่า g(n) อย่างเข้ม
ความสัมพันธ์: o(n) ⊂ O(n) เหมือน < และ ≤ ในคณิตศาสตร์ — little-o เข้มกว่า Big-O
