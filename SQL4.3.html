สิ่งที่ผู้เรียนต้องเข้าใจและทำได้หลังเรียนจบบทนี้

  • เข้าใจแนวคิดและวิธีการพัฒนา ERD และ EERD
  • เข้าใจรูปแบบความสัมพันธ์ที่สำคัญระหว่างข้อมูล

3

แนวคิดสำคัญ (Key Concepts)

แนวคิดพื้นฐานที่ต้องเข้าใจก่อนเริ่มออกแบบ

1. Entity (เอนทิตี)▼

คือวัตถุหลักที่เราต้องการจัดเก็บข้อมูลในฐานข้อมูล แบ่งเป็น:

Strong Entity (เอนทิตีปกติ)

เอนทิตีที่สามารถอยู่ได้ด้วยตัวเอง มีคุณลักษณะที่เป็นเอกลักษณ์ (Primary Key) ได้ด้วยตัวเอง เช่น STUDENT

Weak Entity (เอนทิตีอ่อนแอ)

เอนทิตีที่ไม่สามารถคงอยู่ได้ด้วยตัวเอง ต้องอาศัยการมีอยู่ของเอนทิตีอื่น (Strong Entity) และไม่มีคุณลักษณะหลักที่แท้จริงของตัวเอง เช่น ข้อมูลเวลาเข้าออกงาน (Timestamp) ที่ต้องขึ้นอยู่กับข้อมูลพนักงาน (Employee)

ข้อสังเกต

Weak Entity จะมีเส้นขอบตารางเป็น เส้นประ (Dashed Line) ในแผนภาพ ER

2. Attribute (คุณลักษณะ)▼

คือรายละเอียดหรือคุณสมบัติของเอนทิตี แบ่งออกเป็นหลายประเภท:

Simple vs. Composite

Simple คือแบ่งย่อยไม่ได้แล้ว (เช่น เพศ) ส่วน Composite คือสามารถแยกย่อยได้อีก (เช่น ชื่อ-นามสกุล, ที่อยู่)

Single-valued vs. Multivalued

Single-valued มีได้แค่ค่าเดียว (เช่น วันเกิด) ส่วน Multivalued สามารถมีได้หลายค่าสำหรับ 1 รายการ (เช่น เบอร์โทรศัพท์)

Derived Attribute

คุณลักษณะที่เกิดจากการคำนวณจากข้อมูลอื่น เช่น "อายุ" คำนวณจาก "วันเกิด" ไม่จำเป็นต้องเก็บค่าอายุตรงๆ

Key Attribute (Primary Key)

คุณลักษณะหลักที่มีค่าเป็นเอกลักษณ์ (Unique) ห้ามซ้ำและห้ามเป็นค่าว่าง เพื่อใช้ระบุตัวตนของเอนทิตี (อาจเป็นคีย์เดี่ยวหรือคีย์รวมก็ได้)

3. Relationship (ความสัมพันธ์)▼

คือความเกี่ยวข้องระหว่างเอนทิตี แบ่งตามระดับ (Degree) ได้แก่ Unary (สัมพันธ์ในตัวเอง), Binary (2 เอนทิตี), Ternary (3 เอนทิตี) และแบ่งตามประเภทความแข็งแรงได้แก่:

Weak/Non-Identifying Relationship

ความสัมพันธ์ระหว่าง Strong Entity กับ Strong Entity (PK ของฝั่งหนึ่งไปเป็น FK อีกฝั่ง โดย FK นั้นไม่ได้เป็นส่วนหนึ่งของ PK ใหม่)

Strong/Identifying Relationship

ความสัมพันธ์ระหว่าง Strong Entity กับ Weak Entity (PK ของ Strong จะไปเป็นทั้ง FK และส่วนหนึ่งของ PK ให้กับ Weak Entity)

4. Constraints (ข้อบังคับ/ข้อจำกัด)▼

Connectivity Constraint

อธิบายประเภทความสัมพันธ์หลักๆ คือ 1:1, 1:M, M:N

Cardinality Constraint

บอกปริมาณขั้นต่ำ (Min) และสูงสุด (Max) ของเอนทิตีที่เกี่ยวข้อง รูปแบบคือ (Min, Max)

ตัวอย่างเช่น (0, M) หรือ (1, 1) ข้อมูลฝั่งซ้าย (Min) จะบอกการมีส่วนร่วม ส่วนฝั่งขวา (Max) จะบอก Connectivity

Participation Constraint

  • Partial (ทางเลือก): Min = 0 ไม่จำเป็นต้องมีข้อมูลที่สอดคล้องกัน (Optional) ใช้สัญลักษณ์เส้นเดี่ยว
  • Total (บังคับ): Min = 1 ข้อมูลต้องมีความสัมพันธ์กันอย่างน้อย 1 รายการเสมอ (Mandatory) ใช้สัญลักษณ์เส้นคู่
5. EER Diagram (Extended ER)▼

ใช้แก้ปัญหาฐานข้อมูลที่ข้อมูลมีหลายประเภทย่อย และมีค่าว่าง (NULL) เกิดขึ้นมาก

  • Supertype: เอนทิตีแม่ ที่เก็บ Common Attribute (คุณลักษณะที่ทุกประเภทย่อยมีเหมือนกัน)
  • Subtype: เอนทิตีลูก ที่เก็บ Unique Attribute (คุณลักษณะเฉพาะของประเภทย่อยนั้นๆ)
  • Inheritance: Subtype จะได้รับสืบทอด Primary Key, Common Attributes และ Relationship ทั้งหมดมาจาก Supertype
6. EER Constraints (เงื่อนไขของ EER)▼

Discriminator Constraint

  • Disjoint (D): ข้อมูลใน Supertype จะเป็น Subtype ได้เพียง "ประเภทเดียวเท่านั้น" (Non-overlapping)
  • Overlapping (O): ข้อมูลใน Supertype สามารถเป็น Subtype ได้ "มากกว่า 1 ประเภท" พร้อมกัน

Completeness Constraint

  • Total Completeness (เส้นคู่): ข้อมูล "ทุกแถว" ใน Supertype "ต้อง" เป็นสมาชิกของ Subtype อย่างน้อย 1 ตัว
  • Partial Completeness (เส้นเดี่ยว): ข้อมูลใน Supertype "อาจจะ" เป็นหรือไม่เป็นสมาชิกของ Subtype ก็ได้

สัญลักษณ์ ER Diagram

Chen Notation vs Crow's Foot Notation

Chen Notation

ใช้สัญลักษณ์รูปทรงเรขาคณิต:

  • สี่เหลี่ยม: Entity
  • วงรี: Attribute
  • สี่เหลี่ยมข้าวหลามตัด: Relationship
  • วงรีเส้นคู่: Multi-valued Attribute
  • วงรีเส้นประ: Derived Attribute
  • เส้นประ: Weak Entity

ตัวอย่าง Chen

Student ---- enrolls_in ---- Course
Enrollment_date (Attribute ที่เชื่อมกับ Relationship)

Crow's Foot Notation

ใช้สัญลักษณ์เส้นและอักขระพิเศษ:

  • เส้นเดี่ยว: 1 (One)
  • Crow's Foot: M หรือ N (Many)
  • วงกลมที่ปลายเส้น: Optional (0)
  • เส้นตัด: Mandatory (1)
  • เส้นคู่: Total Participation

2. Cardinality (Min, Max)▼

Cardinality คือช่วงต่ำสุดและสูงสุดของการเกิดข้อมูล เขียนในรูป (Min, Max)

ตัวอย่าง

นักศึกษา 1 คนอาจมีหนังสือ 0 เล่ม หรือสูงสุด 10 เล่ม จะเขียนฝั่งหนังสือเป็น (0, 10)

ข้อควรระวัง - ใส่ฝั่งตรงข้าม!

หากบอกว่า "ลูกค้า 1 คน สั่งสินค้าได้สูงสุด 10 ชิ้น" ค่า (0, 10) จะต้องไปเขียนอยู่ที่ ฝั่งสินค้า ไม่ใช่ฝั่งลูกค้า

ตัวอย่างการวิเคราะห์ Cardinality จากประโยค:

  • โจทย์: "ลูกค้าระบบออนไลน์ 'ทุกคนจำเป็นต้อง' จองสินค้า"
  • ขาไป (ลูกค้าจองสินค้า): ลูกค้า 1 คน จองได้หลายชิ้น (Max = M) และคำว่า "จำเป็นต้อง" หมายถึงห้ามเป็นศูนย์ (Min = 1) ดังนั้นฝั่งสินค้า จะใส่ (1, M)
  • ขากลับ (สินค้าถูกจอง): สินค้า 1 ชิ้น โดนจองโดยลูกค้า 1 คน (Max = 1) แต่สินค้าบางชิ้นอาจจะ 'ไม่มีใครจองเลยก็ได้' (Min = 0) ฝั่งลูกค้า จึงใส่ (0, 1)
3. Participation (การมีส่วนร่วม)▼

Partial Participation

ข้อมูลไม่จำเป็นต้องมี (Min = 0) เช่น สินค้าที่อาจจะยังไม่มีลูกค้าคนไหนมาจองเลย

ในแผนภาพใช้ เส้นเดี่ยว (Single Line)

Total Participation (บังคับ)

จำเป็นต้องมีข้อมูลเกิดขึ้นเสมอ (Min = 1) เช่น ลูกค้า "จำเป็นต้อง" จองสินค้าออนไลน์

ในแผนภาพจะใช้ เส้นคู่ (Double Line)

ข้อควรระวัง - ห้ามขัดแย้ง!

ถ้า Minimum Cardinality เป็น 1 เช่น (1, M) เส้นที่เชื่อมเข้าหาเอนทิตีนั้นจะต้องเป็น เส้นคู่ ถ้า Minimum เป็น 0 เช่น (0, M) เส้นเชื่อมจะเป็น เส้นเดี่ยว ห้ามเขียนขัดแย้งกันเด็ดขาด

EER Diagram (Extended/Enhanced ER)

แบบจำลองขั้นสูง

ในโลกความเป็นจริง ฐานข้อมูลมีความซับซ้อน หากรวมข้อมูลทุกอย่างไว้ใน Entity เดียวจะทำให้เกิดช่องว่าง (NULL Value) จำนวนมาก จึงเกิดแนวคิด EER ขึ้นมา

กระบวนการออกแบบ EER Model▼
  • ขั้นตอนที่ 1: พิจารณา Business Rules - หาคำใบ้จากประโยค เช่น คำว่า "is a type of", "ชนิดหนึ่ง" บ่งชี้ว่าต้องออกแบบเป็น Super/Subtype (EER)
  • ขั้นตอนที่ 2: กำหนด Superclass / Subclass - สร้างโครงร่าง Entity แม่และลูก เช่น Book เป็นแม่ มีลูกเป็น Comic และ Magazine
  • ขั้นตอนที่ 3: หา Common Attribute - ดูคุณลักษณะที่ "ใช้ร่วมกัน" ให้ดึงขึ้นไปไว้ที่ระดับ Superclass
  • ขั้นตอนที่ 4: หา Unique Attribute - ดูคุณลักษณะ "เฉพาะตัว" ให้อยู่ที่ระดับ Subclass
  • ขั้นตอนที่ 5: กำหนด Subtype Discriminator - ระบุแอตทริบิวต์ที่เป็น "ตัวบ่งชี้ประเภท" และค่าที่ใช้แยก
  • ขั้นตอนที่ 6: กำหนด Constraints - ดู Disjoint/Overlap และ Total/Partial Completeness

Constraints ใน EER

สำคัญมาก ออกสอบบ่อย!

เราพิจารณา 2 มิติควบคู่กัน ทำให้เกิดความสัมพันธ์ 4 รูปแบบ (4 Possibilities)

Disjoint (D) - แบ่งแยก

เป็นได้อย่างใดอย่างหนึ่ง ห้ามซ้ำซ้อนกัน เช่น ยานพาหนะ (Vehicle) แยกระหว่าง รถบัส (Bus) กับ เรือ (Boat)

Overlapping (O) - ซ้อนทับได้

เป็นได้มากกว่า 1 อย่างซ้อนทับกัน เช่น Person แยกเป็น Student กับ Employee

Total Completeness (เส้นคู่)

บังคับว่าข้อมูลใน Supertype "ทุกตัว" ต้องลงไปอยู่ใน Subtype อย่างน้อย 1 ตัว

Partial Completeness (เส้นเดี่ยว)

ข้อมูลใน Supertype "บางตัว" อาจจะไม่ตรงกับ Subtype ใดเลยก็ได้

คำถามที่ 1: Vehicle

"พาหนะ (Vehicle) แบ่งออกเป็น รถบรรทุก (Truck) และ รถยนต์ (Car)" โดยพาหนะหนึ่งคันไม่สามารถเป็นทั้งสองประเภทพร้อมกันได้ ควรใช้ Constraint แบบใด?

เฉลย

ใช้แบบ Disjoint (d) เพราะพาหนะ 1 คัน สามารถเป็น Subtype ได้เพียงประเภทเดียวเท่านั้น

คำถามที่ 2: หนังสือและลูกค้า

"หนังสือ 1 เล่มถูกซื้อโดยลูกค้า 1 คนเสมอ (ไม่จำเป็นต้องมีคนซื้อก็ได้) แต่ลูกค้า 1 คน สามารถซื้อหนังสือได้หลายเล่ม หรือไม่ซื้อเลยก็ได้" จงระบุ Cardinality

เฉลย

  • ฝั่งลูกค้า (มองไปหาสินค้า): ซื้อได้ 0 ถึงหลายเล่ม = (0, M)
  • ฝั่งสินค้า (มองไปหาลูกค้า): ถูกซื้อได้ 0 ถึง 1 คน = (0, 1)
  • ทั้งสองฝั่งเป็นแบบ Partial Participation

เฉลย

Subtype จะได้รับสืบทอด Primary Key, Common Attributes และ Relationships ทั้งหมดจาก Supertype เสมอ

คำถามที่ 4: Double Lines

เส้นคู่ (Double Lines) ระหว่าง Superclass และ วงกลม Discriminator มีความหมายว่าอย่างไร?

เฉลย

SQL Cheat Sheet - EER Pattern
-- สร้างตาราง Supertype
CREATE TABLE Supertype (
    ID   INT PRIMARY KEY,
    Name VARCHAR(100),
    Type VARCHAR(20)  -- Discriminator
);

-- สร้างตาราง Subtype (PK = FK ของ Supertype)
CREATE TABLE Subtype_A (
    ID       INT PRIMARY KEY REFERENCES Supertype(ID),
    Unique_A VARCHAR(50)
);

CREATE TABLE Subtype_B (
    ID       INT PRIMARY KEY REFERENCES Supertype(ID),
    Unique_B VARCHAR(50)
);

-- ความสัมพันธ์ M:N ต้องสร้างตารางเชื่อม
CREATE TABLE Junction_Table (
    ID          INT PRIMARY KEY,
    Entity_A_ID INT REFERENCES TableA(ID),
    Entity_B_ID INT REFERENCES TableB(ID)
);

Exam Notes

เคล็ดลับการเก็งข้อสอบ

1. โครงสร้างข้อสอบ Midterm

คะแนนรวม 120 คะแนน (คิดเป็น 35%)

  • ข้อ 1 (20 คะแนน): Principle Concept of Database System (Lecture 1-2)
  • ข้อ 2 (40 คะแนน): Data Model (Lecture 3)
  • ข้อ 3 (60 คะแนน): ERD / EERD (Lecture 4-6) - เป็นข้อที่คะแนนเยอะที่สุด!

กฎการสอบ: นำกระดาษ A4 เข้าห้องสอบได้ 1 แผ่น (หน้า-หลัง) ไม่อนุญาตให้ใช้มือถือ อุปกรณ์อิเล็กทรอนิกส์ หรือพจนานุกรม

2. จุดที่ต้องทำให้ครบ (+ Bonus คะแนน)

  • ระบุ Unique Attribute, Common Attribute ให้ถูกที่
  • เขียน Discriminator Name และ Discriminator Value ให้ครบ
  • ระบุสัญลักษณ์ O หรือ D ในวงกลมให้ถูกต้องตามตรรกะธุรกิจ
  • ขีดเส้นที่วงกลม 1 เส้น (Partial) หรือ 2 เส้น (Total) ให้ถูกต้อง
  • ใส่ Cardinality (Min, Max) ให้ถูกต้อง และใส่เส้นระบุ Participation ให้สอดคล้องกัน
  • ถ้าวิเคราะห์โจทย์แล้วเพิ่ม Attribute ที่สมเหตุสมผลเข้าไปเอง จะได้ คะแนนโบนัส

3. ข้อควรระวัง (Pitfalls) - ห้ามพลาด!

  • สับสนระหว่าง ER และ EER: ถ้ามีคำว่า "is a type of" ให้ใช้ EER ทันที
  • ตีความ Cardinality ผิดฝั่ง: ค่า (Min, Max) ต้องใส่ที่ฝั่งตรงข้าม
  • Participation ขัดแย้งกับ Cardinality: ถ้า Min = 1 เส้นต้องเป็นคู่ ถ้า Min = 0 เส้นต้องเป็นเดี่ยว
  • วาง Attribute ผิดที่: Common Attribute ต้อง "เลื่อนขึ้น (Promoted)" ไปอยู่ชั้น Supertype
  • Inheritance: ห้ามลืมว่า Subclass จะได้รับ Primary Key มาจาก Superclass เสมอ
