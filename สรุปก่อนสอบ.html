ข้อมูลมีการยกระดับคุณค่าตามการประมวลผล 4 ขั้น

📦

Raw Facts — ข้อเท็จจริงดิบที่ยังไม่ผ่านการประมวลผล เช่น วันเกิด, น้ำหนัก

↓ ผ่านการประมวลผล
📊

ข้อมูลที่คำนวณ/สรุปแล้ว เช่น อายุ (ปีปัจจุบัน - ปีเกิด), BMI (น้ำหนัก/ส่วนสูง²)

↓ วิเคราะห์รูปแบบ
🧠

สารสนเทศที่วิเคราะห์หา Pattern ทางสถิติ เช่น "ลูกค้าที่ซื้อน้ำ 50% มักซื้อขนมปังด้วย"

↓ ตัดสินใจ
⚡
Wisdom (ปัญญา)

การตัดสินใจของมนุษย์จากความรู้ — ไม่จัดเก็บในคอมพิวเตอร์

ข้อสอบออกแน่!

  • จะไม่จัดเก็บ Information ไว้ใน DB เพราะมันแปรผัน (Dynamic) เปลี่ยนแปลงตลอดเวลา (เช่น อายุเปลี่ยนทุกปี) — ระบบ UI จะดึง Data มาคำนวณแล้วแสดงเป็น Information
  • Wisdom ไม่บันทึกใน DB เพราะขึ้นอยู่กับวิจารณญาณมนุษย์
  • Data Object ต้องแยกให้ถูก เช่น "อาการไอ" ไม่ใช่คอลัมน์ของผู้ป่วย แต่เป็น Object ใหม่ (โรค/อาการ) ที่ต้องแยกเป็นอีกตาราง เพราะผู้ป่วยหลายคนแชร์อาการเดียวกันได้

1.2 ปัญหาของระบบแฟ้มข้อมูล (File System Problems)

ก่อนมีฐานข้อมูล เราเก็บข้อมูลแยกเป็นไฟล์ (เช่น Excel แยกระหว่างฝ่าย) ทำให้เกิดปัญหา 4 อย่าง:

ปัญหาความหมายตัวอย่าง
Structural Dependenceโปรแกรมต้องรู้โครงสร้างไฟล์ เปลี่ยนทีต้องแก้โค้ดทั้งหมดเปลี่ยนความยาวตัวอักษร → ต้องแก้โปรแกรมใหม่
Data Dependenceเปลี่ยน Data type → โปรแกรมพังทันทีเปลี่ยน Text เป็น Integer → โค้ดเดิมใช้ไม่ได้
Island of Informationข้อมูลแยกกันอยู่ สร้างรายงานรวมยากExcel ฝ่ายบุคคลกับฝ่ายขายแยกกัน
Data Redundancyข้อมูลซ้ำซ้อน → เกิด Data Anomaliesข้อมูลลูกค้าซ้ำในหลายไฟล์

Data Anomalies 3 แบบ (ต้องจำ!)

  • Insertion Anomaly: เพิ่มข้อมูลไม่ได้ — เช่น ยังไม่มีลูกค้าคนไหนสั่งซื้อสินค้า → ใส่ข้อมูลสินค้าใหม่ไม่ได้ในตารางคำสั่งซื้อ
  • Modification Anomaly: แก้ไขแล้วข้อมูลไม่สอดคล้องกัน — เช่น เปลี่ยนที่อยู่ลูกค้า 1 คน แต่ข้อมูลลูกค้าคนเดียวกันในอีกตารางยังเป็นที่อยู่เดิม
  • Deletion Anomaly: ลบข้อมูลแล้วข้อมูลสำคัญอื่นหายไปด้วย — เช่น ลบคำสั่งซื้อสุดท้าย ทำให้ข้อมูลสินค้าหายไปด้วย

Excel ไม่ใช่ Database!

Excel เป็นเพียงโปรแกรมตารางคำนวณ (Computerized file systems) ไม่มีคุณสมบัติ DBMS ในการจัดการข้อจำกัดโครงสร้างข้อมูลระดับลึก

1.3 ระบบฐานข้อมูล (Database Systems) และ DBMS

องค์ประกอบของ Database Architecture 5 อย่าง

  • Hardware: เครื่องคอมพิวเตอร์, Server, Storage
  • Software: OS, DBMS (MySQL, Oracle), Application Programs
  • People: DBA, นักออกแบบ, ผู้ใช้ปลายทาง
  • Procedures: กฎเกณฑ์การใช้งานและดูแลระบบ
  • Data: ข้อมูลที่จัดเก็บในระบบ

หน้าที่ของ DBMS (7 อย่าง)

  • จัดการพจนานุกรมข้อมูล (Data Dictionary)
  • บริหารพื้นที่จัดเก็บ
  • ดูแลความปลอดภัย (Security)
  • จัดการการเข้าถึงพร้อมกัน (Concurrency Control)
  • รักษาความสมบูรณ์ของข้อมูล (Data Integrity)
  • จัดการ Backup & Recovery
  • ให้บริการ SQL (Structured Query Language)

ประโยชน์ของ DBMS

  • ลดความซ้ำซ้อนของข้อมูล (Data Redundancy)
  • ข้อมูลสอดคล้องกัน (Data Consistency)
  • มีความปลอดภัย (Security)
  • ความเป็นอิสระของข้อมูลและโครงสร้าง (Data & Structural Independence)
  • จัดการข้อมูลได้อย่างมีประสิทธิภาพ

Data Structures 3 ประเภท

  • Structured: มีรูปแบบชัดเจน (เช่น ตาราง SQL)
  • Semi-structured: มีบางส่วนที่เป็นระเบียบ (เช่น XML, JSON)
  • Unstructured: ไม่มีรูปแบบ (เช่น รูปภาพ, วิดีโอ, อีเมล)

2.2 4 Basic Building Blocks

องค์ประกอบพื้นฐานของ Data Model

🏢
Entity (วัตถุข้อมูล)
วัตถุที่ต้องการจัดเก็บข้อมูล แปลงมาจาก คำนามในประโยค
Physical คน, สถานที่, สินค้า
Logical การลงทะเบียน, การสอบ, กำไร, อุบัติเหตุ
📋
Attribute (คุณลักษณะ)
คุณสมบัติของ Entity = Column ในตาราง
เช่น ID, Name, Email — แปลงมาจากคำคุณศัพท์
🔗
Relationship (ความสัมพันธ์)
เชื่อมโยง Entity 2 ตัวขึ้นไป แปลงมาจากคำกริยา
.types: 1:1, 1:M, M:N
⚖️
Constraint (ข้อบังคับ)
เงื่อนไขบังคับ มักใช้คำว่า "ต้อง (must)"
เช่น "นักศึกษาต้องลงทะเบียน 5 วิชา"

ข้อสอบออกแน่!

  • Logical/Intangible Entity จับต้องไม่ได้แต่เป็น Entity ได้! เช่น "การลงทะเบียน" (Enrollment) เป็น Process ที่เป็น Entity ได้
  • คำว่า "must" = สัญญาณของ Constraint ในประโยค Business Rule
  • Data Semantic = ความหมายของข้อมูลที่แฝงอยู่ — ต้องมี Data Dictionary เพื่ออธิบายให้เครื่องเข้าใจ

2.3 Business Rules & Forward/Backward Analysis

การวิเคราะห์ความสัมพันธ์ — ต้องทำทั้ง 2 ขาเสมอ!

การแปลง Business Rule

  • คำนาม (Noun) → แปลงเป็น Entity
  • คำกริยา (Verb) → แปลงเป็น Relationship
  • "ต้อง (must)" → แปลงเป็น Constraint

Forward Analysis (Active Voice — ขาไป)

"Student attend classes" → 1:M (นักเรียน 1 คน เข้าเรียนได้หลายคลาส)

Backward Analysis (Passive Voice — ขากลับ)

"Classes are attended by students" → 1:M (คลาส 1 ห้อง ถูกเรียนโดยนักเรียนหลายคน)

สรุป: Forward 1:M + Backward 1:M = M:N (Many-to-Many)

รวมกัน = ความสัมพันธ์จริงคือ Many-to-Many

ข้อควรระวัง!

ห้ามวิเคราะห์เพียงฝั่งเดียว! ถ้าโจทย์ให้มาขาเดียว ต้องคิดย้อนกลับเองเสมอ ส่วนใหญ่ความสัมพันธ์จริงคือ M:N

ตัวอย่าง: Author writes Book

โจทย์บอก: "หนังสือแต่งโดยผู้เขียนหลายคน" (Passive 1:M)

คิดย้อนกลับ: "ผู้แต่ง 1 คน ก็เขียนหนังสือได้หลายเล่ม" (Active 1:M)

สรุป → M:N

3.2 ชนิดของ Entity

Entity แบ่งออกเป็น 2 แกน: แกนที่ 1 แบ่งตามความเป็นเจ้าของ (Strong vs Weak) และแกนที่ 2 แบ่งตามบทบาท (Master vs Transaction)

ประเภทความหมายสัญลักษณ์ตัวอย่าง
Strong Entityอยู่ด้วยตนเองได้ มี PKสี่เหลี่ยม 1 ชั้นStudent, Product, Order
Weak Entityต้องพึ่งพา Strong Entityสี่เหลี่ยม 2 ชั้น (เส้นประ)Cancel_Order, Dependent
Master Entityวัตถุหลักในระบบ ไม่ได้เกิดจาก Process—Student, Product
Transaction Entityเกิดจากกระบวนการทางธุรกิจ—Borrow, Return, Enrollment

Strong Entity vs Weak Entity

Weak Entity

ตัวอย่าง: Withdraw (การถอนรายวิชา)

→ เป็น Weak Entity เพราะต้องมีการลงทะเบียนก่อน

→ เป็น Transaction Entity เพราะเป็นกระบวนการ

ดังนั้น Withdraw = Weak + Transaction ในเวลาเดียวกัน

  • ตัวเลข (Min, Max) เขียนไว้ที่ฝั่งตรงข้ามของ Entity ที่มันอธิบาย!
    เช่น "ลูกค้า 1 คน สั่งสินค้าได้สูงสุด 10 ชิ้น" → (0,10) เขียนที่ฝั่งสินค้า ไม่ใช่ฝั่งลูกค้า
  • ห้ามขัดแย้ง! ถ้า Min = 1 (Mandatory) → เส้นต้องเป็นเส้นคู่ ถ้า Min = 0 → เส้นเดี่ยว ห้ามใส่ 0 แต่ขีด 2 เส้น (โดนหักคะแนนทันที)

Participation Constraint

Optional / Partial (Min = 0)

ไม่จำเป็นต้องมี เส้นเดี่ยว
เช่น สินค้าที่อาจยังไม่มีลูกค้าจอง

Mandatory / Total (Min = 1)

บังคับต้องมี เส้นคู่ (NOT NULL)
เช่น ลูกค้า "ต้อง" จองสินค้าออนไลน์

Relationship ตาม Degree

  • Unary: Entity สัมพันธ์กับตัวเอง เช่น พนักงานดูแลพนักงานด้วยกัน
  • Binary: 2 Entity เช่น พนักงานทำงานให้แผนก
  • Ternary: 3 Entity เช่น หมอสั่งจ่ายยาให้คนไข้
11

3.5 Chen Notation vs Crow's Foot Notation

Chen Notation

  • สี่เหลี่ยม: Entity
  • วงรี: Attribute
  • สี่เหลี่ยมข้าวหลามตัด: Relationship
  • วงรีเส้นคู่: Multi-valued
  • วงรีเส้นประ: Derived
  • เส้นประ (กรอบ): Weak Entity

Crow's Foot Notation

  • เส้นเดี่ยว: 1 (One)
  • Crow's Foot: M หรือ N (Many)
  • วงกลมที่ปลายเส้น: Optional (0)
  • เส้นตัด: Mandatory (1)
  • เส้นคู่: Total Participation
12

3.6 EER Diagram (Extended/Enhanced ER)

แบบจำลองขั้นสูง — ลดความซ้ำซ้อนและค่า NULL

ขั้นตอนการสร้าง EER

  1. กำหนด Supertype/Subtype (Data Cluster)
  2. หา Common Attributes → ไว้ที่ Supertype
  3. หา Unique Attributes → ไว้ที่ Subtype
  4. กำหนด Discriminator (ตัวแบ่งแยก) ลงในวงกลม
  5. กำหนด Constraints 2 มิติ (Disjoint/Overlap + Total/Partial)

ตัวอย่าง EER: บัญชีธนาคาร

Account (Supertype)
Name, Surname, Balance
D

Current Account
Overdraft Limit

EER Constraints 4 แบบ (ออกสอบ!)

Constraintความหมายสัญลักษณ์
Disjoint (D)ข้อมูลเป็น Subtype ได้เพียง ประเภทเดียวเท่านั้น (ห้ามซ้ำ)D ในวงกลม, เส้นเดี่ยว
Overlapping (O)ข้อมูลเป็น Subtype ได้ มากกว่า 1 ประเภท พร้อมกันO ในวงกลม, เส้นเดี่ยว
Total Completenessข้อมูลทุกแถวใน Supertype ต้องปรากฏใน Subtype อย่างน้อย 1เส้นคู่ (double line)
Partial Completenessข้อมูลใน Supertype อาจจะไม่ปรากฏใน Subtype เลยก็ได้เส้นเดี่ยว (single line)

ความหมายของ Supertype / Subtype

  • Supertype (ตัวแม่): เก็บ Common Attribute ที่ใช้ร่วมกัน
  • Subtype (ตัวลูก): เก็บ Unique Attribute เฉพาะของแต่ละประเภท + สืบทอด PK, Attributes, Relationships ทั้งหมดจากตัวแม่

จุดออกสอบ (Exam Traps)

ข้อผิดพลาดที่มักเกิดในข้อสอบ — ระวังให้ดี!

Trap 1: Physical vs Logical Entity

ข้อสอบหลอกด้วยสิ่งนามธรรม/กระบวนการ เช่น "Enrollment", "Examination" → คำตอบคือเป็น Logical Entity ได้!

Trap 2: Forward/Backward Analysis

ให้ประโยคมาทางเดียว แล้วถามความสัมพันธ์จริง → ต้องวิเคราะห์ Passive Voice ย้อนกลับเสมอ เพื่อเช็คว่าเป็น M:N หรือไม่

Trap 3: แปลง Business Rule เป็น ERD

ให้ Statement มา แล้วถามว่าคำใดเป็น Entity/Relationship → ต้องแยก Noun = Entity, Verb = Relationship

Trap 4: Schema vs Instance

Schema = โครงสร้าง (Column Name) ส่วน Instance = ข้อมูลดิบ (Raw Data) ที่อยู่ข้างใน

Trap 5: Cardinality ใส่ผิดฝั่ง

ตัวเลข (Min, Max) ต้องเขียนที่ฝั่งตรงข้ามของ Entity ที่อธิบาย ไม่ใช่ฝั่งเดียวกัน

Trap 6: Derived Attribute ไม่มีตัวตั้งต้น

วาด Derived Attribute (เช่น BMI) แต่ไม่มี Attribute ตั้งต้น (Weight, Height) → ผิด!

Trap 7: Participation ขัดแย้งกับ Cardinality

ใส่ Min = 1 แต่ขีดเส้นเดี่ยว หรือใส่ Min = 0 แต่ขีดเส้นคู่ → โดนหักคะแนนทันที
